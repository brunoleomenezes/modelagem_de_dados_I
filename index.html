Atividade em Grupo — Projeto de Banco de Dados

Disciplina: Modelagem de Dados I (MD I)

Objetivo

Construir, em grupo, um projeto de banco de dados completo, passando pelas etapas:

  1. Levantamento de requisitos.
  2. Modelo Entidade-Relacionamento (DER).
  3. Modelo Relacional (MR).
  4. Implementação em SQL no OneCompiler PostgreSQL.
  5. Execução de consultas.

Etapas da Atividade

1. Levantamento de Requisitos

  • Identificar os principais atores do sistema (clientes, funcionários, produtos, pedidos, etc.).
  • Listar os dados que precisam ser armazenados.

2. Modelo Conceitual (DER)

  • Construir o diagrama em dbdiagram.io, draw.io ou papel.
  • Incluir entidades, atributos, relacionamentos e cardinalidades.

3. Modelo Relacional (MR)

  • Migrar o DER para tabelas relacionais.
  • Definir tabelas, atributos, chaves primárias e estrangeiras.
  • Normalizar até a 3FN.

4. Implementação em SQL

  • Criar tabelas no OneCompiler PostgreSQL com CREATE TABLE.
  • Definir tipos de dados, PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE.
  • Inserir ao menos 5 registros por tabela.

5. Consultas SQL

Criar pelo menos 5 consultas. Exemplos:

  • Listar todos os clientes.
  • Mostrar pedidos feitos em uma data.
  • Encontrar o produto mais vendido.
  • Calcular total de vendas de um cliente.
  • Listar serviços de um funcionário específico.

Cenário 1 — E-commerce de Eletrônicos "TechStore RJ"

Contexto

A TechStore RJ é uma loja virtual de eletrônicos (celulares, notebooks, periféricos). O banco deve organizar clientes, pedidos, estoque, entregas e pagamentos.

Requisitos

  • Clientes: nome, CPF, e-mail, telefones, endereços.
  • Produtos: código, nome, descrição, preço, marca, categoria, estoque, desconto.
  • Pedidos: data, status, forma de pagamento; cada cliente pode ter vários pedidos.
  • Pagamentos: data, valor, status, info do meio (cartão, pix, boleto).
  • Entregas: transportadora, rastreio, prazos, status.

Regras

  • Estoque reduz quando o pedido é confirmado.
  • Pedido precisa ter ao menos 1 item.
  • Cliente pode ter até 3 endereços de entrega.
  • Entrega só ocorre após pagamento.

Exemplo de Dados

  • Cliente João Silva, CPF 123.456.789-10, 2 telefones, 1 endereço.
  • Produto Notebook Dell Inspiron 15, R$ 3.500, estoque 20.
  • Pedido 02/09/2025, status pago, cliente João.
  • Itens: 1 notebook + 2 mouses sem fio.
  • Pagamento: cartão Visa, final 1234, valor R$ 3.960.
  • Entrega: Correios, rastreio BR123456, entregue 05/09/2025.

Cenário 2 — Clínica Médica "Vida & Saúde"

Contexto

A clínica oferece consultas médicas e exames. O banco deve organizar pacientes, médicos, consultas, receitas, exames e faturamento.

Requisitos

  • Pacientes: dados pessoais, convênio opcional, histórico de alergias.
  • Médicos: nome, CRM, especialidade, contatos, agenda.
  • Consultas: data, hora, paciente, médico, sala, status.
  • Receitas: ligadas à consulta, medicamentos com dosagem.
  • Exames: tipo, data, resultado, médico responsável.
  • Faturamento: valor, forma de pagamento (dinheiro, cartão, convênio), status.

Regras

  • Paciente pode ter várias consultas, cada uma com 1 médico.
  • Uma consulta pode gerar múltiplos exames.
  • Receita está sempre vinculada à consulta.
  • Pagamentos podem ser particulares ou convênio.
  • Exames só após consulta.

Exemplo de Dados

  • Paciente Maria Souza, CPF 987.654.321-00, convênio Unimed, carteirinha 11223344.
  • Médico Dr. Carlos Mendes, CRM 12345, cardiologista.
  • Consulta 10/09/2025 às 14h, status realizada.
  • Receita: Dipirona 500mg, 1 comprimido a cada 8h por 5 dias.
  • Exame: Eletrocardiograma, resultado normal, 11/09/2025.
  • Pagamento: convênio Unimed, autorizado, valor R$ 150.

Entregáveis

  1. DER (imagem ou desenho).
  2. Modelo Relacional (tabelas e chaves).
  3. Script SQL no OneCompiler (CREATE + INSERT).
  4. Consultas SQL com resultados.
  5. Relatório em PDF ou apresentação curta (5-10 min).

Critérios de Avaliação

  • Clareza e organização do DER (20%).
  • Correção do Modelo Relacional e normalização (20%).
  • Implementação em SQL correta (20%).
  • Qualidade das consultas SQL (20%).
  • Apresentação e defesa do projeto (20%).
